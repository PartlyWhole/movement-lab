LESSON first — what does it mean for a box to move?
M1 MEET POSITION x += k

Nothing actually moves. x is just a number — where to draw the box this frame. To make it move, you add a fixed step to x every frame, right inside the game loop. Press STEP to run one frame: x grows by k each time, so the box slides at a steady pace. Movement is just repeated addition.

STEP — ADD k TO x EACH FRAME
◢ THE MATH
x += k  # every frame, in the loop
in plain words → you never compute a position from a formula — you accumulate it, adding the same step each frame. After n frames you've landed at n·k, but you got there by adding, not multiplying. That running total is the box's memory — and everything else in this lab just changes what you add.
M2 MEET VELOCITY x += vx

In M1 we added the same fixed step every frame. Now we store that step in its own variable — velocity vx, the change-in-x-per-frame — so we can change it. Make it negative to go left, zero to stop, big to go fast. Same x += vx as before; the difference is that now the amount you add is a thing you control (and, next panel, a thing that can change).

EACH STEP ADDS vx TO x
◢ THE MATH
x = x + vx after n frames: x = x₀ + n·vx
in plain words → velocity is how far you move per frame. A constant velocity draws a straight line: same step, every frame, forever.
M3 MEET ACCELERATION vx += ax

One more link in the chain. If velocity can change position, what changes velocity? Acceleration ax — change-in-velocity-per-frame. Gravity is nothing more than a constant ax. This is a position-vs-frame graph: with acceleration on, velocity ramps up and the line curves into a parabola. STEP and watch ax → vx → x.

POSITION (DOWN) vs FRAME (RIGHT) — IT CURVES
◢ THE MATH
vx = vx + ax x  = x  + vx
in plain words → a two-link chain run every frame: acceleration nudges velocity, velocity nudges position. Constant velocity is a straight line; constant acceleration bends it into a curve.

        01
        CONSTANT VELOCITY
        pos += vel
      

      

        
The simplest mover there is, and the one you'll write most. Add velocity to position once per frame and the object slides at a steady speed — bullets, scrolling backgrounds, conveyor belts.

        

          
◢ THE MATH

          
posₙ = posₙ₋₁ + vel

          
in plain words → velocity is pixels-per-frame. Constant velocity means equal steps forever — a straight line through space.

        02
        ACCELERATION & GRAVITY
        vel += acc
      

      

        
Add a constant downward acceleration each frame and straight-line motion bends into a parabola — the arc of every jump and thrown object. Gravity is just acc that never turns off.

        

          
◢ THE MATH

          

            vel += acc  then  pos += vel
            peak height = v² / (2·g)
          

          
in plain words → velocity changes by a fixed amount each frame, so position changes by a growing amount — that's curvature. Launch speed alone sets how high you reach.

        03
        FRICTION / DAMPING
        vel *= k
      

      

        
Multiply velocity by a number just under 1 every frame and it decays toward zero — a smooth, natural slide to a stop. The closer k is to 1, the longer the glide. This is "ice physics" and momentum-feel in one line.

        

          
◢ THE MATH

          

            vel *= k  (0 < k < 1)
            total slide ≈ v · k / (1 − k)
          

          
in plain words → each frame you keep a fraction k of your speed. Geometric decay: never quite zero, but quickly close enough to snap to a halt.

        04
        CLAMP / TERMINAL VELOCITY
        max(-m, min(m, vel))
      

      

        
Gravity left alone accelerates forever — soon your object falls faster than it can be drawn. Clamp velocity to a maximum and it settles at a steady terminal velocity. The same trick caps run speed, dash speed, anything.

        

          
◢ THE MATH

          
vel = max(-m, min(m, vel))

          
in plain words → min(m, vel) stops it exceeding +m; max(-m, …) stops it dropping below −m. Acceleration keeps adding, but the cap holds.

        05
        IMPULSE — THE JUMP
        vel = -J
      

      

        
A jump isn't a force you hold — it's a single frame where you slam velocity to a big negative number. Gravity (panel 02) does all the rest. Press JUMP, then STEP through the arc. Jump again mid-air for a double / triple jump — each impulse is smaller than the last.

        

          
◢ THE MATH

          

            on jump:  vel = -J
            multi-jump: J · 0.7ⁿ for the nth jump
          

          
in plain words → an impulse is an instant change of velocity. Set it once; let the per-frame gravity + position update carry it through a clean arc.

        06
        SMOOTHING / LERP
        pos += (target - pos) * t
      

      

        
Move a fraction t of the remaining gap each frame: fast when far, gentle when close. This is the easing behind smooth camera-follow, UI tweens, and "lazy" turrets. Move the target, then STEP and watch pos glide in.

        

          
◢ THE MATH

          
pos += (target - pos) * t

          
in plain words → the step size is the gap times t, so it shrinks as you approach — exponential ease-out, no easing curve table required.

      THE FINE PRINT
      
      two truths nobody warns you about
    


    
    

      

        F1
        FRAME vs SECOND
        pos += vel * dt
      

      

        
A frame is not a fixed amount of time. pos += vel means "per frame" — so on a 30 FPS machine your game moves half as fast as on 60 FPS. Multiply by dt (seconds since last frame) and velocity becomes "per second": every machine agrees. STEP the clock and watch the naive movers drift apart while the * dt movers stay locked together.

        

          
◢ THE MATH

          

            naive:  pos += vel     # fps-dependent
            fixed:  pos += vel * dt # fps-independent
          

          
in plain words → 60 FPS does twice as many updates per second as 30 FPS. Without dt that means twice the distance; with it, each update is scaled down to match.

        F2
        FLOAT vs INTEGER
        rect.x = round(fx)
      

      

        
Remember the Rect from Collision Lab? Its x and y are integers — they can't hold half a pixel. Store velocity straight into the Rect and a sub-pixel speed like 0.7 gets truncated to 0 every frame: the box never moves (or stutters). Keep the true position in a float and only assign rect.x = round(fx) when you draw.

        

          
◢ THE MATH

          

            int:   rect.x = int(rect.x + vel)  # loses the fraction
            float: fx += vel; rect.x = round(fx)
          

          
in plain words → the Rect is for collision and drawing — integers, on purpose. The physics lives in floats. Round only at the very last moment.
